## Closing the C64 AI development loop

RPI (Reflect, Plan, Implement) changed the way I prompt and the way the agent works. Instead of asking for a giant code dump, I run a longer loop: reflect on what just happened in the last build, plan a small targeted change, then implement and immediately rebuild and run. For this to work propperly I need the AI CLI agent to edit files, invoke KickAssembler, and inspect the resulting output and symbols. In practice, "lengthening the loop" means fewer prompt rewrites and better results.

As described in part 2 I chose AI CLI tooling. This tooling allowed to work with a tight build-run-debug loop. OpenCode is a valid choice for this project because it combines direct file edits, terminal execution, and model switching in one place, so the same agent that writes 6502 can also assemble, inspect logs/symbols, and adjust based on real output. 

### Skills: Teaching the AI the rules of the 80's

One of the key concepts in OpenCode is the use of **skills**, reusable instruction sets you attach to a project that teach the AI the rules of *your* specific world. Think of them as onboarding documentation for a new hire who happens to know every programming language ever written but has never heard of the Commodore 64. Without them, every session starts from scratch. For this project, three skills were created:

**c64-assembler** is the foundation. It encodes everything specific to writing 6502 KickAssembler code for *this* codebase: the full memory map (`$2000` for the custom charset, `$3000` for sprite bitmaps, `$6000` for the game entry point), the zero-page ownership rules (so the scroll engine doesn't accidentally trash the logo drawing code), naming conventions (`snake_case` labels, `SCREAMING_SNAKE_CASE` constants, module prefixes like `bg_`, `sfx_`, `kuro_`), hardware rules (the read-modify-write pattern for hardware registers, IRQ save/restore sequences, raster-safe screen write windows), and KickAssembler specifics like `BasicUpstart2` and sprite pointer math. Without this skill, the AI would confidently produce code that compiles fine in the abstract but crashes the moment it lands on real C64 hardware, wrong addresses, clobbered zero-page, bad timing. With it loaded, the AI produces code that already understands the constraints of this specific machine and this specific codebase.

**c64-sidmusic** encodes what makes authentic 1980s SID music actually sound like 1980s SID music. The SID chip has three voices, no true polyphony, and a shared filter, and the composers who made it legendary worked around those limits with specific techniques: fast arpeggios to simulate chords, pulse-width modulation for that evolving tone, vibrato and hard restarts for expressiveness, and filter sweeps for atmosphere. This skill gives the AI all of that context in structured form: voice roles, waveform choices, tempo ranges, the stylistic fingerprint of composers like Jeroen Tel and Ben Daglish, and, critically, the failure modes to avoid (no modern synth layering, no polyphonic chords, respect the timing constraints). The result is that instead of generating something that sounds vaguely like a ringtone, the AI has a real chance of producing something that sounds like it belongs on a C64.

**vice-c64-probe** is where things get genuinely interesting, because this skill is what actually *closes the loop*. The first two skills make the AI better at writing code. This one makes it possible for the AI to *verify* that the code works. The skill connects OpenCode to the VICE C64 emulator via its Binary Monitor Protocol over TCP. After KickAssembler produces a `.prg` file, the agent can automatically launch VICE, load the program, let it run for a few seconds, and then query the live emulator state: CPU registers, zero-page contents, VIC-II sprite registers, SID register values, sprite pointer tables, all of it, returned as structured JSON the agent can reason about. This means the AI no longer has to guess whether the sprite pointer at `$07F8` resolves to the right address, or whether the zero-page variables landed where expected, or whether the VIC-II screen enable bit actually got set. It can check. The build, run, observe, fix loop that previously required a human to open VICE, inspect memory, and feed the results back manually now happens automatically within a single agent session. That is what "closing the loop" means in practice: the AI writes code, assembles it, runs it on the actual (emulated) hardware, reads back the hardware state, and uses that evidence to decide what to fix next, without a human in the middle.

This shift to a closed loop is subtle but important. The feedback loop tightens considerably when the tool can read the current state of the memory output of the emulator. This made the whole process alot faster and more reliable.

## Building the Scrolling Background
Ok back to the game, i wanted this game to resemble something like FIST II and Usagi Jojimbo which at least meant a scrolling background. This is where things get genuinely interesting from a technical standpoint and i hoped the AI could realy help be out here.

### The C64 Scrolling Problem

The C64 has no dedicated scrolling hardware like later machines, so games such as FIST II and Usagi Yojimbo had to have the CPU manually shift everything on screen. The trick is timing: the screen is drawn top to bottom like an old TV, and there's a brief moment at the top of each frame before the visible picture starts where you can safely update the screen without causing a glitch. Miss that window and you get a visible tear where the old and new images overlap.

### The Approach: A Character-Based Ring Buffer

The solution the AI came up with ,and that I went with, is a **character-based ring buffer**. The idea is straightforward: which is a fixed-size loop of tile data 64 columns wide. The visible screen is 40 columns wide, so scrolling just means moving a read pointer forward or backward through that loop. When it reaches the end it wraps back to the start, like this:

```
Ring buffer (64 tiles wide):
[ col 0 | col 1 | ... | col 39 | col 40 | ... | col 63 ]
          ^--- viewport (40 cols) ---^
          scroll_offset = 1
```

Every time the player moves, the whole visible screen (960 character cells) is redrawn from that offset. No copying, no math heavier than a wraparound mask. The scroll is not pixel-smooth, it jumps one full character (8 pixels) per step. The jiffy clock slows it down enough that it does not feel jarring, but it is coarser than what you would see on more capable hardware.

### The Charset: Tiles as Characters

This is where the "character-based" part becomes important. The C64's character mode does not draw pixels directly, it draws *characters*, where each character is an 8×8 pixel bitmap defined in a charset. By replacing the ROM charset with a custom one and pointing `$d018` at it, every byte written to screen RAM becomes a tile index rather than a text glyph. 

The game's background charset lives at `$2800` (`game_bg_charset` in `gfx.asm`) and is set active by writing `$1a` to `$d018` at game start. The charset is defined in **multicolor mode** (`$d016` bit 4 set), which gives each 8×8 cell four colors: a global background color, two global shared colors (`$d022`/`$d023`), and one per-cell color from color RAM. That last one is the per-row flat color from `bg_row_color`.

The tile set contains 18 background tiles, each a single byte index written to screen RAM. I asked the AI to create a background with distand hills, some green fields , flowers a road and some grass. It came up with:

| Index | Tile | What it looks like |
|-------|------|--------------------|
| `$00` | `TILE_SKY` | Empty background color |
| `$01` | `TILE_FAR_HILL` | Distant hill bump |
| `$02–$05` | `TILE_NEAR_HILL_*` | Near hill left/peak/right/fill |
| `$06` | `TILE_FIELD` | Green field with flower specks |
| `$07` | `TILE_ROAD` | Road surface |
| `$08–$09` | `TILE_GRASS_TOP_A/B` | Grass blade variants |
| `$0a` | `TILE_GRASS_FILL` | Solid grass |
| `$0d–$0e` | `TILE_TREE_TOP_L/R` | Tree canopy |
| `$19` | `TILE_PILLAR` | Temple pillar |

The tile map itself (`bg_tile_map` in `game.asm`) is 25 rows × 64 columns of byte-sized tile indices. Rows 0–5 are sky, rows 6–7 tree canopy, rows 8–9 hills, rows 10–15 field and road, rows 16–24 grass. The AI generated this layout and it looks reasonable, a side-scrolling karate game needs sky, terrain, and a road, and that is what it produced. The distant hill looked more like trees in my opinion, so i added some tree stumps to make it look somewhat better. 

### Is This the Right Approach?

Honestly: probably not the most technically elegant approach for the C64, and I was not fully convinced when the AI suggested it.

The classic C64 technique for smooth horizontal scrolling uses the VIC-II fine scroll register to shift 0–7 pixels for free, and only redraws one column of screen RAM per frame (the column that just scrolled into view). That keeps the CPU load down and gives proper sub-character-cell smoothness. Apperently games like Turrican use variations of this. The character-redraw-ring-buffer approach is simpler to reason about and to generate with AI, there is no column tracking, no partial update, no fine scroll bookkeeping. But it rewrites all 960 cells every scroll step, which is heavy, and the 8-pixel-per-step coarseness is noticeable.

The AI went with the ring buffer approach because it is conceptually clean and straightforward to implement correctly. I went along with it because at this stage of the experiment, getting *something* working and visible mattered more than getting it pixel-perfect. Whether this needs to be revisited in a later part is an open question. So i was a bit disapointed that my scrolling background didn't look as good as i had in mind, but went with it anyway.

### The Raster-Safe Window

The one area where the AI did need guidance was timing. The initial generated scroll code wrote to screen RAM without any raster synchronisation, which works fine in a slow loop but causes visible tearing as the VIC-II and the CPU race each other across the same memory.

The fix was a `wait_frame_safe_window` routine that waits for the raster to reach the bottom border (`$d012 >= $f8`), then waits for it to wrap through line 256 and come back below line `$20`. That returns control during the top border, around 30 raster lines where the VIC-II is not reading the visible area. The entire 960-cell redraw is wrapped in `sei`/`cli` and executed inside that window. It is tight but it fits.

The AI understood the concept when prompted with the right framing. What it could not do on its own was verify that the timing actually held, that the redraw finished before line `$20`. That required running the code in VICE and watching for tearing, which is exactly the kind of feedback loop the `vice-c64-probe` skill was built for.

## Gameplay: Making it Actually a Game

A scrolling background is a tech demo. A game needs a player, obstacles, and something to lose. Here is where the gameplay loop came together, and where the AI had to start holding several things in its head simultaneously.

### Player Sprite Animations

The player sprite has 14 distinct frames spread across walk cycles, special poses, and damage states:

- **Walk**: 3 frames per direction (left/right), cycling at 6 jiffy ticks per frame (~0.12 seconds each). A full walk cycle takes about 0.36 seconds.
- **Kick, kneel, jump, hit**: single static frames each, no sub-animation, just a pose held for a fixed duration.

The animation is driven by the jiffy clock at `$a2` (the PAL 50 Hz counter that the KERNAL increments via interrupt). The walk routine reads `$a2`, computes elapsed ticks since the last frame change, counts down a timer, and advances `anim_frame` (mod 3) when it expires. The current direction (left or right) determines which of the two frame tables (`anim_ptrs_right` / `anim_ptrs_left`) to index into, and the result is written directly to `$07f8`, the VIC-II sprite 0 pointer register.

Special poses are managed by `pose_mode`, a simple state variable with five values:

| Pose | Trigger | Duration |
|------|---------|----------|
| Normal | — | Continuous |
| Kick | Space | ~1 second |
| Kneel | S | ~1 second |
| Jump | W | ~4.5 seconds (arc) |
| Hit | Enemy contact | ~0.16 seconds |

While any special pose is active, movement input is completely ignored, the main loop checks this and skips straight back to the top. That one `bne game_main_loop` after `update_pose_state` is all that separates "I pressed kick" from "I can also walk at the same time", and it makes the combat feel deliberate rather than twitchy.

The jump is the most involved: a 32-entry signed-offset table drives the arc over 25 airborne ticks, then lands. During the air phase the player can still scroll the background (or move in arena mode), and projectiles pass straight through, kneeling dodges knives, jumping dodges everything.

The AI generated all of this in reasonable shape. The frame table structure and jiffy-timer pattern are things it handles well. Where it needed correction was the jump arc: the first attempt produced a parabola that went up and never came back down, because the signed byte subtraction on the Y register was not being handled correctly. One build-inspect-fix cycle sorted it.

### Keyboard Input

The C64 keyboard is not a simple set of individual lines, it is a matrix, and reading it correctly requires writing a column selection mask to `$dc00` (CIA port A) and then reading the row result from `$dc01` (CIA port B). The logic is active-low: a zero bit means the key is pressed. Each key-check routine wraps the read in `sei`/`cli` to prevent the IRQ from interfering with `$dc00` mid-scan.

The key layout for the game:

| Key | Action |
|-----|--------|
| D | Move right / scroll right |
| A | Move left / scroll left |
| Space | Kick |
| W | Jump |
| S | Kneel / duck |

The AI got CIA scanning right on the first attempt, likely because it is a well-documented and frequently reproduced pattern. The only edge case was Space on the title screen, which uses the KERNAL `GETIN` routine instead of direct CIA scanning. That distinction was already in the `c64-assembler` skill, so no surprises there.

## The Improved AI-Assisted Development Experience: An Honest Assessment

At this point in the experiment I have three blog posts worth of data on what it actually feels like to "vibe code" 6502 assembly with AI assistance. Here is the honest scorecard.

### What Genuinely Got Better

- **Context persistence via skills**, no longer re-explaining the memory map every session.
- **Agentic file editing**, the AI modifies files directly instead of dumping walls of code into a chat window.
- **Model choice**, switching to a stronger model for a particularly gnarly routine (e.g. the scroll engine timing) and back to a faster one for boilerplate is now a one-line config change.
- **Closed-loop debugging**, with the VICE probe skill active, the agent can assemble, run, and inspect hardware state without a human in the middle. Catching a wrong sprite pointer or a misconfigured VIC-II register went from a multi-minute manual inspection to a single agent step.

### What Still Requires Human Judgment

- **Raster timing**, the AI understands the concept but still occasionally produces scroll code that tears. Catching this requires knowing what tearing *looks like* in VICE, which the AI cannot see.
- **Memory conflicts**, zero-page allocation is manual. The AI will happily reuse `$fb–$fe` in game code without knowing that `main.asm` owns that range during the intro.
- **"Looks right, runs wrong"**, 6502 is unforgiving. A single off-by-one in a pointer dereference corrupts whatever memory happens to live nearby. The AI produces these bugs at roughly the same rate as a junior human developer, which is both reassuring and alarming.
- **Sprite art is not a solved problem for generic AI**, this one deserves its own paragraph, because it is where the gap between "AI can do this" and "AI can do this *well*" is most visible. Asking a general-purpose AI model to generate C64 sprite data from scratch produces results that range from bad to completely unrecognisable. The constraints are extreme: 24×21 pixels, four colours maximum in multicolor mode, each pixel actually two bits wide. A generic model has no real concept of what those constraints *look like* at the end, so it fills the bytes with patterns that are technically valid but visually meaningless, blobs, noise, or something vaguely humanoid if you squint. Kuro, the level 1 boss, fell into the "bad but recognisable" category and I left him in. For the level 2 dragon, Lung, the generic AI output was so far off that I had to abandon that path entirely and use a dedicated pixel art tool, PixelLab, to design the sprite as actual art, then convert it through Spritemate into C64 format. The results were immediately and obviously better. The lesson is that sprite art on constrained hardware is a *visual design problem first*, and generic language models are not visual designers. Dedicated tools that understand the pixel grid, the colour limits, and the aesthetic of 8-bit art produce sprites that look like they belong on a C64. General AI produces sprites that look like they belong in a bug report.

### The Bigger Picture

So where does this experiment actually land? My honest conclusion after three parts is this: **it is possible to build a working C64 game with AI assistance, but it is very difficult to build one that feels like it belongs to the 1980s.**

The gap is not primarily technical. The AI can write correct 6502 assembly, manage sprite state machines, implement IRQ handlers, and wire up CIA keyboard scanning without embarrassing itself. With the right skills and a closed build loop, it does the *engineering* work at a level that would have seemed remarkable a few years ago. What it struggles with is the aesthetic and intuitive layer, the part that made those original games feel alive. The smooth pixel scrolling, the sprite art that reads instantly at low resolution, the SID tune that gets stuck in your head, these things required their creators to deeply understand both the machine and the craft of making something beautiful within brutal constraints. They worked within those limits *expressively*. The AI works within them *correctly*, which is not the same thing.

The scrolling in this game jumps one full character per step. The sprite art for the bosses needed human-directed tools to look reasonable. The music is serviceable but not memorable. None of that is a failure of the AI exactly, it is more a reflection of what "vibe coding" at the 6502 level actually means in practice. You get a game. You do not automatically get a *good* game. The 1980s originals were made by people who ate, breathed and dreamed this hardware. No amount of prompt engineering fully substitutes for that.

What AI *does* provide is a remarkably powerful way to explore a domain you would otherwise never be able to enter at all. I could not have written this game without it. Every line of assembly in this codebase would have taken me weeks of study to produce on my own, if I ever managed it at all. With AI, I spent that time on the parts that actually required my judgment: what should the game *be*, how should it *feel*, what is worth fixing and what is good enough. That is a genuinely different and interesting division of labour, and I think it has real value, even if the end result would not pass for a commercial release from 1987.
